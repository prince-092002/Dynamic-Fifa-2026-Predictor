# Phase 5G — Leakage-Safe Model Diagnostics, Feature Optimization & Challenger Evaluation

Generated by `src/modeling/phase5g_diagnostics.py` · dataset: 49,589 international matches (1872-11-30 → 2026-07-04) · chronological 70/15/15 split · seed 42 · runtime ~7 min · **no new dependencies**.

## Executive summary

The current production **XGBoost model was faithfully reproduced** (test accuracy 0.6075, macro-F1 0.4511, log loss 0.8607, Brier 0.5056 — matching the dashboard exactly) and **remains the best validated production model**. A bounded, leakage-safe search over features, recency weighting, class weighting, and 30 hyperparameter configurations produced **no challenger that legitimately beats it**: the best tuned challenger scored test accuracy 0.6069 (a −0.0005 change; 95% bootstrap CI [−0.0023, +0.0012], i.e. indistinguishable from zero) with slightly worse log loss. **Decision: REJECT — production XGBoost retained.** This is the honest, expected result for an Elo-dominated football-outcome model already near its data ceiling.

The headline diagnostic finding — *why XGBoost's macro-F1 (0.451) is lower than Logistic Regression's (0.527) despite better accuracy/log loss/Brier* — is fully explained by the **draw class** and is a **metric artifact, not a defect** (details below).

## 2. Why XGBoost macro-F1 < Logistic Regression macro-F1 (calculated, not speculated)

Draws are the **minority class** (22.7% of training rows; 1,703 of 7,439 test rows) and the intrinsically hardest outcome to call. On the test set:

| | Predicted draws | Draw recall | Draw precision | Draw F1 | Macro-F1 |
|---|---:|---:|---:|---:|---:|
| Logistic Regression | 1,658 | 0.287 | 0.294 | **0.290** | 0.527 |
| XGBoost | **46** | **0.012** | 0.457 | **0.024** | 0.451 |

- **XGBoost predicts "draw" only 46 times out of 7,439** — because a draw is almost never the single most likely outcome (argmax) of a calibrated 3-way probability. Its draw *F1* is therefore ~0.02, which drags the unweighted macro-F1 down to 0.451.
- **Logistic Regression uses `class_weight="balanced"`**, which up-weights the minority draw class in the loss, so it argmax-predicts ~1,658 draws (recall 0.287, F1 0.290) and lifts macro-F1 to 0.527 — **but pays for it everywhere that matters**: lower accuracy (0.575 vs 0.608), worse log loss (0.887 vs 0.861), worse Brier (0.524 vs 0.506), and ~9× worse calibration (ECE 0.046 vs 0.0053).
- **XGBoost is superbly calibrated.** Its confidence bands track accuracy almost exactly: predictions at 0.37 conf are right 37.6% of the time; 0.55→54.4%; 0.65→64.9%; 0.75→75.0%; 0.88→88.0%.

**Conclusion:** macro-F1 is the wrong headline metric for this system. The model's output feeds a **Monte Carlo simulator that samples from probabilities** — it never needs the model to argmax "draw". XGBoost still assigns a well-calibrated ~26% draw probability per match; those draw probabilities are used correctly downstream. The low macro-F1 reflects hard-class argmax behavior on a minority class, not a probabilistic weakness. Selecting XGBoost by validation log loss was the correct decision.

## 5. Experiments (all evaluated on the chronological validation set; test set untouched)

| Experiment | Val acc | Val macro-F1 | Val log loss | Val Brier | Verdict |
|---|---:|---:|---:|---:|---|
| E0 baseline (prod features + params) | 0.5824 | 0.4301 | 0.8978 | 0.5293 | reference |
| A — multi-window form (+11 last_3/last_10 + draw_rate diffs) | 0.5820 | 0.4283 | 0.8974 | 0.5289 | no value (last_5 already captures signal) |
| D — recency weight, half-life 8y | 0.5811 | 0.4432 | 0.9015 | 0.5308 | ↑macro-F1 but ↓log loss → reject |
| D — recency weight, half-life 2y | 0.5787 | 0.4586 | 0.9090 | 0.5352 | ↑macro-F1, ↑draw recall, but ↓accuracy & ↓calibration → reject |
| **G — inverse-frequency class weights** | 0.5354 | **0.5127** | 0.9348 | 0.5546 | draw recall 0.01→0.39 but accuracy −0.047, log loss +0.037 → **reject (kills calibration)** |
| G — draw boost ×1.8 | 0.5406 | 0.5014 | 0.9257 | 0.5487 | same tradeoff → reject |
| F — best of 30 hyperparameter configs | 0.5827 | 0.4303 | 0.8978 | 0.5293 | ≈ baseline (already near-optimal) |

**Feature-family ablation (validation):** Elo-only already reaches 0.576 acc / 0.918 log loss (Elo = ~53% of feature importance). Each family (form, goals, h2h, tournament, schedule) adds small but real incremental value; the **full 25-feature production set is best** on both validation accuracy (0.582) and log loss (0.898). No family is pure noise — nothing should be removed. Full table: `feature_ablation_results.csv`.

**Key experiment insight — the draw/calibration tradeoff is now quantified:** you *can* lift macro-F1 from 0.43 to 0.51 by class-weighting to chase draws, but it costs 4.7 accuracy points and materially worsens log loss (0.898→0.935) and Brier — i.e. it degrades exactly the probability quality the Monte Carlo simulator depends on. Correctly rejected.

## 8. Promotion decision — REJECT (production XGBoost retained)

Gates: meaningful+significant accuracy gain (≥0.005 and CI>0) → **fail**; probability quality preserved → pass; CV stability → pass. Not all gates cleared, so **no model change is forced**, per Phase 5G rules. The current 60.75% XGBoost remains the production model. Recorded in `promotion_decision.json`. **No downstream regeneration was performed** (production model unchanged → live predictions, Monte Carlo outputs, champion/finalist probabilities, and completed-match locking are all untouched).

## 9. Calibration

XGBoost test **ECE = 0.0053** (near-perfectly calibrated; 9× better than Logistic Regression's 0.0463). No post-hoc calibration (isotonic/sigmoid) is warranted — it would risk making a well-calibrated model worse. This is the strongest single justification for keeping XGBoost as the simulator's probability source.

## 10. Limitations & recommended future work

- **Draw predictability is data-limited**, not model-limited: draws cluster in close (low-Elo-gap) matches that are inherently near-random. No leakage-safe feature tested moved draw *probability quality* meaningfully.
- **Not implemented (require feature-pipeline extension over raw history, out of this phase's bounded scope):** Elo-momentum features (Elo slope over last N) and opponent-strength-adjusted form (avg opponent Elo over last N). The training dataset already contains multi-window form/goals columns (used in Experiment A) but not momentum/opponent-adjusted aggregates. These are the most promising untried directions and are the natural first step of a future modeling phase.
- **Alternative libraries** (LightGBM, CatBoost, Optuna) are not installed; adding them was avoided to preserve deployment simplicity. A custom bounded randomized search was used instead.
- **Recommended next phase:** Phase 5H — Player, Squad & Tactical Intelligence (as planned), plus optionally revisiting Elo-momentum / opponent-adjusted form via a controlled feature-pipeline extension. Do **not** chase macro-F1 via class weighting — this phase proved it degrades simulator-relevant probability quality.
